# PRD：可观测 AI 自动化运行时（Automation Runtime）

## 1. 产品概述

### 1.1 产品名称（暂定）
- 中文：可观测 AI 自动化运行时
- 英文：Observable AI Automation Runtime

### 1.2 一句话定义
把脆弱的网页自动化脚本，升级成可观测、可维护、可持续优化、可被 AI 协同诊断与调优的自动化执行系统。

### 1.3 产品定位
这是一个面向复杂网页流程的自动化执行引擎，不是单纯录制点击的 RPA 工具，也不是纯聊天式 AI 助手。它的核心价值在于：
- 将复杂网页流程拆成结构化阶段
- 为每个阶段提供统一结果契约
- 让自动化运行结果可被人和 AI 诊断、比较、优化、归档和交付

### 1.4 当前最贴切的产品形态
- 下层：Automation Runtime（执行层）
- 上层：Brain / Agent / Assistant（诊断层、调优层、编排层）

当前产品更适合定位为：
- 复杂网页自动化的执行内核
- AI-native automation runtime
- 可观测浏览器自动化基础设施

## 2. 背景与问题定义

### 2.1 现有市场痛点
当前市面上的浏览器自动化方案，常见问题包括：

1. 脚本能跑，但不可维护
- 逻辑耦合严重
- 页面一变就整条链返工
- 局部问题难以局部修复

2. 自动化是黑盒
- 只知道“成功/失败”
- 不知道卡在哪一步
- 不知道为什么慢
- 日志杂乱，缺乏阶段化语义

3. 批量运行后结果不清晰
- 缺乏 latest / batch archive / summary 的统一语义
- 难以做跨批次对比
- 难以追踪失败模式和性能变化

4. AI 很难真正接入自动化运行时
- 缺少结构化字段
- 缺少可供诊断的 artifacts
- 缺少明确的阶段边界与状态契约
- 导致 AI 只能“看日志猜问题”，不能稳定参与调优

### 2.2 本产品要解决的问题
本产品要解决的不是“网页能不能自动点”，而是：
- 自动化如何长期稳定运行
- 失败和瓶颈如何清晰定位
- 批量运行结果如何结构化交付
- AI 如何真正参与自动化诊断、调优和维护

## 3. 目标用户

### 3.1 核心目标用户

#### A. 自动化重度用户 / 运营型用户
典型特征：
- 有长期重复网页流程
- 需要稳定批量跑任务
- 需要结果归档、交付、追踪
- 对失败可解释性敏感

#### B. 技术团队 / 自动化工程团队
典型特征：
- 使用 Playwright/Puppeteer/Browser automation
- 需要更强的工程化能力
- 希望自动化可维护、可扩展、可诊断

#### C. AI 自动化探索者 / Agent 系统构建者
典型特征：
- 想让 AI 参与自动化诊断与调优
- 需要结构化的执行内核作为 Agent 的专业后端
- 希望把 AI 和自动化运行时结合

### 3.2 非核心用户
- 只想做简单点击录制的普通用户
- 只需要一次性脚本、不关心长期维护的个人用户

---

## 4. 产品目标

### 4.1 短期目标
1. 将复杂网页流程标准化为 staged architecture
2. 形成统一的阶段结果契约
3. 提供批量运行、latest、summary、archive 体系
4. 提供足够强的失败定位与性能分析能力

### 4.2 中期目标
1. 将 AI 诊断、调优能力接入运行时
2. 把自动化执行层与大脑层解耦
3. 形成面向多站点、多场景的 adapter 体系
4. 形成可开源的 core runtime 和可私有的策略层分层

### 4.3 长期目标
1. 成为 AI Agent 的专业自动化执行后端
2. 形成 Automation Brain + Runtime 双层系统
3. 形成适合商业化的自动化基础设施平台

---

## 5. 核心价值主张

### 5.1 对用户的价值
#### 价值 1：自动化不再是黑盒
用户可以明确知道：
- 当前运行到哪个阶段
- 哪一步慢
- 哪一步失败
- 为什么失败
- 是否可以继续优化

#### 价值 2：自动化从脚本升级为系统
通过阶段拆分、结果契约、artifact 归档，自动化从“一次性脚本”升级为“可维护系统”。

#### 价值 3：让 AI 真正能参与自动化
结构化状态、失败原因、阶段边界和 detail 使 AI 可以稳定参与：
- 故障诊断
- 参数调优
- 文档对齐
- 维护和收口

#### 价值 4：批量运行结果可交付
运行结束后，用户拿到的不是一堆杂乱日志，而是：
- latest 视图
- batch summary
- per-batch archive
- session/account/delivery 结构化产物

## 7. 核心功能设计

### 7.1 Staged Automation Runtime
将复杂流程拆成明确阶段，例如：
- proxy-precheck
- entry
- credential-submit
- verification-submit
- profile-completion-submit
- post-auth-ready
- account-delivery

每一阶段具备：
- 独立阶段职责
- 独立 adapter 边界
- 统一结果输出
- 独立失败分类与日志

### 7.2 阶段结果契约
每个阶段输出统一结构：
- success
- stage
- state
- reason
- nextStage
- signalStrength
- settleStage
- detectionSource
- stateChanged
- retryCount
- detail

该结构可供：
- 运行时推进
- 人工诊断
- AI 消费
- 批次归档

### 7.3 Batch Runner
支持：
- 并发 worker
- 账号队列
- 代理分配
- per-run summary
- latest 视图
- archive 持久化

### 7.4 Artifact 体系
系统需要稳定产出：
- latest summary
- batch summary
- stage results
- session records
- delivery payload
- failure samples

### 7.5 AI 协同层
对 AI 暴露稳定输入：
- stage results
- batch artifacts
- field semantics
- failure taxonomy
- performance baselines

AI 可输出：
- 诊断结论
- 调优建议
- 策略建议
- 文档收口建议

## 8. 产品优势

### 8.1 竞争优势
1. 比普通脚本更强的可观测性
2. 比普通 RPA 更强的复杂网页工程化能力
3. 比纯 Agent 自动点网页更稳定、更可维护
4. 更适合 AI 作为诊断脑、调优脑、编排脑接入

### 8.2 技术壁垒
1. staged architecture 拆分能力
2. 阶段结果契约设计
3. artifact 归档和批次语义
4. session / delivery / post-auth 语义沉淀
5. 失败模式收口和优化路径积累

---

## 9. 劣势与关键风险

### 9.1 产品劣势
1. 不属于“一眼就能被大众秒懂”的产品
- 需要 demo 和包装解释价值

2. 工程成本较高
- 阶段边界、字段契约、detail 分层都需要长期维护

3. 初期不如简单脚本上手快
- 用户需要理解 staged 模型才能真正发挥价值

### 9.2 商业风险
1. 容易被误解成“某站自动注册工具”
- 伤害产品形象
- 增加敏感风险

2. 如果走错定位，容易和通用 agent 平台硬碰硬
- 如 OpenClaw、通用助手平台
- 不利于建立差异化

3. 若过早全量开源真实策略层，可能削弱商业壁垒

### 9.3 技术风险
1. 文档契约落后于代码能力
2. detail 字段若无限膨胀，会削弱稳定性
3. 若上层 AI 直接绕开框架自由点击，会破坏 staged 优势

## 10. 市场需求判断

### 10.1 市场存在性
市场是存在的，而且在增长。

原因：
1. 浏览器自动化需求本来就长期存在
2. 纯脚本方案维护成本越来越高
3. AI 大模型兴起后，市场开始期待“AI + 自动化”
4. 但现有方案往往缺少真正适合 AI 的结构化执行层

### 10.2 市场空白
当前市面主流更多是：
- 普通 Playwright/Puppeteer 脚本
- 录制型 RPA
- 工作流编排系统
- 纯 Agent 浏览器操作

而“阶段化、结构化、可归档、可优化、适合 AI 消费”的自动化运行时，不是主流形态，存在明显产品空白。

### 10.3 市场接受度判断
#### 对普通大众
直接接受度一般，需要强包装。

#### 对技术团队/自动化团队
接受度高，只要 demo 足够直观。

#### 对 AI agent / automation builder 群体
接受度会更高，因为他们最能理解可观测运行时的价值。

## 11. 盈亏点分析

### 11.1 收入机会点
#### 方向 A：企业/团队版订阅
卖点：
- 稳定运行
- 批量执行
- 可诊断
- 结果可交付
- AI 调优能力

#### 方向 B：专业执行引擎授权
作为：
- 内部自动化内核
- Agent 平台的浏览器执行后端
- 私有化部署模块

#### 方向 C：高级能力增值
可收费能力：
- 高级诊断分析
- AI 调优建议
- 性能 benchmark
- 高级 artifact dashboard
- 私有 knowledge pack

#### 方向 D：定制化实施/咨询
基于 staged runtime 为企业落特定站点或复杂流程

### 11.2 成本点
#### 技术成本
- 维护 staged core
- adapter 开发
- 文档契约收口
- AI 协同层开发
- artifacts / dashboard / benchmark 基础设施

#### 产品成本
- 教育市场成本高
- 需要 demo 和案例
- 需要较强文档与产品包装能力

#### 风险成本
- 开源边界把控
- 敏感场景合规边界
- 平台化与垂直化取舍

### 11.3 盈亏关键点
是否能盈利，关键不在“功能多不多”，而在：
1. 能否把自己包装成“自动化基础设施”，而不是“脚本工具”
2. 能否抓住技术团队和 AI automation builder 这批高价值用户
3. 能否保留框架层开放、策略层私有的商业边界
4. 能否尽快做出让人一眼看懂的 demo

## 12. 竞争格局简析

### 12.1 与普通 Playwright/Puppeteer 工程流对比
优势：
- 更强可观测性
- 更强结构化结果
- 更适合批跑、归档、AI 消费

### 12.2 与传统 RPA 对比
优势：
- 更适合复杂网站
- 更适合工程团队维护
- 阶段语义更强

### 12.3 与纯 Agent Browser 对比
优势：
- 更稳定
- 更可诊断
- 更适合长期运行
- 不把成功建立在 AI 临场自由点击上

### 12.4 与 OpenClaw / Claude 这类平台对比
不应正面定义为同类。
更健康的关系应是：
- OpenClaw / Claude：大脑 / 平台 / 编排层
- 本产品：专业自动化执行层

## 13. 产品策略建议

### 13.1 推荐主方向
不要把产品主叙事放在：
- 自动注册
- 批量脚本
- 某站流程打通

推荐主叙事：
- 可观测自动化
- AI-native automation runtime
- 自动化执行基础设施

### 13.2 推荐产品路线
#### 第一阶段
把 staged runtime 做稳：
- 阶段契约
- 文档收口
- artifact 体系
- benchmark

#### 第二阶段
接入 AI 诊断与调优：
- diagnosis brain
- tuning brain
- maintenance brain

#### 第三阶段
对接上层平台：
- OpenClaw
- Claude
- 其他 agent runtime

#### 第四阶段
做开源 core + 私有策略层

## 14. 开源与商业边界建议

### 14.1 适合开源的部分

## 16. 总结

这是一个有明显差异化潜力的产品方向。

它的最大价值，不是“自动做网页流程”，而是：
- 让自动化变得可观测
- 让自动化可以长期维护
- 让自动化可以被 AI 理解和优化
- 让自动化结果可以被结构化沉淀和交付

### 成功关键
- 定位不能跑偏成“脚本工具”
- 架构契约必须继续收口
- demo 必须让价值一眼可见
- 商业边界要切清楚：框架层 vs 策略层

### 最值得坚持的方向
把它做成：
**复杂网页自动化的 AI-native runtime，而不是另一个黑盒自动化工具。**
