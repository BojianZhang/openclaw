# Head First 设计模式 - 深摘要

## 这本书解决什么问题
它解决的是：**当代码开始因为变化而混乱时，如何用更稳的对象协作方式把变化隔离开。**

## 核心主线
1. 识别变化点
2. 把变化封装起来
3. 通过组合、委托、多态降低耦合
4. 用模式描述常见可复用解法

## 最关键的知识骨架
### 1. 模式的本质不是模板，而是“对变化的回应”
真正值得使用模式的前提是：
- 某个点将来会变
- 这个变化已经让代码痛苦
- 你希望把变化限制在局部

### 2. 优先组合而不是继承
这是整本书非常重要的设计倾向。
继承常常让变化传播得太广，而组合更适合按需拼装行为。

### 3. 模式要和场景绑定理解
例如：
- 策略模式：算法/规则会变化
- 观察者模式：状态变化要广播
- 工厂模式：创建逻辑复杂或要隐藏实现
- 装饰器模式：行为要可叠加
- 适配器模式：接口不兼容但想复用

## 工程落地价值
### 适合回答的问题
- 为什么这里该用策略模式？
- 什么时候该用工厂，什么时候直接 new？
- 装饰器和继承有什么区别？
- 设计模式到底怎么落到业务代码里？

### 在项目中最有用的地方
- 复杂条件分支的收敛
- 可切换规则的建模
- 插件式扩展点设计
- 外部系统接入适配层

## 常见误区
- 误把模式当“高级语法”
- 没有变化点也硬套模式
- 一上来设计过度，导致代码更绕

## 与其他书的连接
- 和《重构》一起用：一个告诉你可用的结构，一个告诉你怎么渐进迁移过去
- 和《Effective Java》一起用：在模式使用之外，约束类和接口设计质量
