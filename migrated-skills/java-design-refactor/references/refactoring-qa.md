# 问答模板：重构

## Q: 老代码太乱了，应该怎么重构？
### 短答版
不要一上来重写，先识别坏味道，再小步重构：提炼函数、拆分类职责、收敛条件分支、持续验证。

### 展开版
重构的正确顺序通常是：
1. 先识别问题：重复代码、长函数、巨型类、条件泥团、职责混乱
2. 先改最痛的点：提炼函数、提炼对象、搬移职责
3. 每次改动保持行为不变
4. 每改一小步就验证

重构的目的不是“写得更优雅”，而是让系统更容易改、出问题时影响更可控。

## Q: 什么时候该重构，什么时候该重写？
### 短答版
大多数情况下优先重构；只有当系统边界已经完全错误、维护成本远高于重建，才考虑重写。

### 展开版
如果代码还能跑、业务还在变、历史知识还重要，那么重构通常比重写更稳。重写最大的问题不是开发，而是丢失历史经验和引入新 bug。
