# Java 入门级 Web 项目答题模板

当用户要讲传统 JSP/Servlet/DAO 风格的 Java 入门项目时，优先使用这份模板。

## 1. 你做过什么项目

### 短答版
我做过一个 Java 入门级 Web 项目，核心是一个云日记系统，采用 JSP + Servlet + MySQL + Tomcat 的传统 B/S 架构，用 Maven 管理项目。功能上主要包括用户登录、免登录、用户模块、类型模块、云记模块、主页展示和统计报表。这个项目虽然不是微服务，但很适合体现我对传统 Java Web 开发流程、数据库设计、前后端交互和模块拆分的理解。

### 展开版
如果让我介绍，我会先强调它是一个经典的 Java Web 单体项目，而不是微服务系统。整体技术栈比较传统：JSP 负责页面展示，Servlet 负责请求处理，MySQL 存储业务数据，Tomcat 负责运行，Maven 做项目构建。
从功能上看，项目是按业务模块逐步长出来的：先是用户登录和免登录，再是用户信息模块、类型模块、云记模块，然后补主页展示和统计报表。这里最重要的不是用了多少中间件，而是通过这个项目把 MVC 思维、DAO 封装、Session/Cookie 登录态处理、前后端数据交互、数据库设计这些基础能力跑通了。

---

## 2. 这种项目的价值怎么讲

### 回答思路
这类项目的价值不在于架构复杂，而在于它能很好体现基础功。比如：
- 功能怎么拆模块
- 登录态怎么处理
- BaseDao 怎么做数据库访问封装
- 页面和后端怎么交互
- 报表和统计功能怎么组织
如果把这些基础讲清楚，反而会比硬讲中间件更真实。

---

## 3. 为什么 BaseDao 这种内容值得讲

### 回答思路
因为它体现的是最基础的数据访问抽象能力。入门项目里如果每个模块都直接手写 JDBC，会很乱；BaseDao 这种封装虽然不复杂，但它体现了“把重复代码下沉”的意识，这对后面理解 ORM、Repository、数据层分离都很重要。

---

## 4. 为什么这种项目不该硬讲成微服务

### 回答思路
因为问题域不一样。它本来就是一个单体 Web 入门项目，价值在于传统 Web 开发流程、模块划分、数据库和前后端交互。如果硬套成微服务/分布式项目，会显得不真实，也会把项目真正有价值的基础能力讲丢。

---

## 5. 项目连接句
- “这个项目虽然不复杂，但很适合体现我对传统 Java Web 基础能力的掌握。”
- “我会重点讲功能模块拆分、登录态处理、BaseDao 封装、数据库设计和前后端交互，而不是硬讲成分布式架构。”
- “如果面试官继续追问，我会把它当成一个基础能力样本，而不是强行拔高成复杂平台项目。”
