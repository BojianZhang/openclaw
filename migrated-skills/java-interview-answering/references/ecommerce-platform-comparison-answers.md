# 电商平台对照答题模板

当用户需要对比 乐SHOP 和 易购商城 这两类 Java 商城平台样本时，优先使用这份模板。

## 1. 这两套项目怎么对比

### 短答版
如果简化来说，我会把两套项目都看成“模块化电商平台”的样本，它们都稳定地拆出了 common、portal、manager、sso、order、rpc 这类边界。区别在于，乐SHOP 更适合讲平台能力逐步扩展，比如 Redis、Elasticsearch、支付、短信、云存储这些能力是怎么按阶段进入的；而 易购商城 更适合讲传统 Java 商城路线，比如 Dubbo + ZooKeeper、Solr、购物车、订单支付、注册验证和 RabbitMQ 这条经典演进链。

### 展开版
我会先讲相同点，再讲差异点。相同点是两套项目都说明了一个事实：不管技术栈怎么换，电商平台里 common、portal、manager、sso、order、rpc 这些边界是比较稳定的。
差异点在于，乐SHOP 更像一个偏 Spring Boot 风格的模块化平台演进样本，强调平台能力怎么一步步加上 Redis、ES、支付、短信、云存储；而 易购商城 更像传统 Java 商城项目样本，Dubbo + ZooKeeper 的协作方式更明显，Solr 搜索、购物车、订单支付、注册验证和 MQ 的教学路径也更典型。
所以如果面试官问平台能力演进，我更愿意引用乐SHOP；如果问传统商城架构、Dubbo、Solr、购物车/订单链，我会更愿意引用 易购商城。

---

## 2. 为什么两套项目都拆 common / portal / manager / sso / order / rpc

### 回答思路
因为这些边界更多是由电商业务和平台角色决定的，而不是由某个具体中间件决定的。用户侧需要 portal，后台运营需要 manager，统一身份需要 sso，交易真相需要 order，共享支持需要 common，跨模块协作需要 rpc。技术栈会变，但这些平台边界不会轻易消失。

---

## 3. 乐SHOP 和 易购商城 各自更适合讲什么

### 回答思路
乐SHOP 更适合讲平台能力引入顺序和模块化平台演进，比如 Redis、ES、支付、短信、云存储什么时候进入比较合理。易购商城更适合讲传统 Java 商城架构，包括 Dubbo + ZooKeeper、Solr 搜索、购物车、订单支付、注册验证和 RabbitMQ 这些经典能力怎么串起来。
