# 电商平台项目追问模板

当用户已经能讲清“做了一个模块化电商平台”，但还需要接更深的项目追问时，优先使用这份模板。

## 1. 为什么 portal 和 manager 要分开

### 回答思路
因为两者面对的角色、流程和目标完全不同。portal 面向终端用户，核心是商品浏览、购物体验和用户交互；manager 面向后台运营，核心是商品管理、内容维护、订单管理、运营配置。虽然都属于 Web 系统，但如果长期混在一个模块里，后续会把面向用户的逻辑和面向运营的逻辑搅在一起，边界会越来越乱。

---

## 2. 为什么要单独做 SSO

### 回答思路
因为登录、身份、Session/Token 这类能力通常会被多个业务模块复用，不适合散落在 portal 或 order 里。单独做 SSO，可以让身份能力更统一，也更方便结合 Redis 这类高频状态访问能力做会话管理和快速校验。

---

## 3. 为什么 Redis 是在那个阶段引入的

### 回答思路
我会说 Redis 不是为了让技术栈看起来更丰富，而是因为平台到了那个阶段，热点读取、状态访问、Session/Token 访问这类问题已经开始明显影响体验和数据库压力，所以才把 Redis 作为一个平台支撑能力引进来。它解决的是高频访问压力，而不是替代核心业务域。

---

## 4. 为什么 Elasticsearch 不是一开始就上

### 回答思路
因为 ES 的价值主要体现在搜索体验，而不是平台基础可用性。一开始先把平台、商品、基本购物链跑通更重要；当商品检索、过滤、搜索体验开始成为真实需求时，再引入 ES 会更合理。否则很容易出现平台还没稳定，就先把搜索体系搞复杂。

---

## 5. 为什么 order 和 payment 要分开讲

### 回答思路
order 是平台内部的业务真相，负责订单创建、查询和状态流转；payment 是外部支付能力接入，负责把外部支付结果映射回平台内部订单状态。两者高度相关，但不是同一层边界。如果混着讲，就会把内部业务状态和外部支付交互混成一团。

---

## 6. 为什么支付不是一开始就做

### 回答思路
因为支付依赖于前面的订单能力已经比较稳定。只有当平台已经有明确的订单创建和状态流，支付接入才有清晰落点。否则支付只是接了 SDK，但内部业务真相还没稳定，后续会非常难维护。

---

## 7. 短信验证为什么算支撑能力，不是主域

### 回答思路
因为短信通常服务于登录、注册、敏感操作确认等流程，本质上是身份验证或风控辅助能力。它很重要，但通常是围绕核心业务流程提供保障，而不是独立决定平台主业务边界。

---

## 8. 这个项目里最重要的架构取舍是什么

### 回答思路
我会说最重要的不是“用了多少技术”，而是模块边界和能力引入顺序。比如先把 portal、manager、SSO、order 这些核心边界讲清楚，再把 Redis、搜索、支付、短信、云存储当作围绕这些边界逐步引入的支撑能力。这个顺序比技术清单更能体现架构判断。

---

## 9. 项目答辩连接句
- “我会把这个项目讲成一个按业务边界和能力压力逐步演进的平台，而不是技术堆砌。”
- “如果面试官追问，我会先解释为什么这么拆模块，再解释为什么在那个阶段引入 Redis、搜索、支付、短信这些能力。”
- “这个项目最值得讲的不是用了多少中间件，而是核心边界和支撑能力之间的关系。”
