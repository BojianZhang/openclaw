# 电商平台项目答题模板

当用户要讲 Java 电商平台项目，尤其是包含门户、后台、SSO、订单、Redis、搜索、支付、短信这类能力的项目时，优先使用这份模板。

## 1. 你做过什么电商项目

### 短答版
我做过一个模块化电商平台项目，整体上分成几个比较清晰的能力边界：面向用户侧的门户系统、面向运营侧的后台管理系统、单点登录能力、订单能力，以及公共支持模块和 RPC 服务契约层。在这个基础上，再逐步引入 Redis 做缓存优化、Elasticsearch 做搜索能力、支付集成、短信验证和云存储等能力。我的理解是，这类项目的重点不是单独用了哪些中间件，而是怎么围绕真实购物流程把能力边界逐步长出来。

### 展开版
如果让我介绍这个项目，我会先从模块边界讲，而不是直接罗列技术栈。
用户侧有 portal，负责商品浏览、购物流程和用户交互；后台有 manager，负责管理端运营；登录相关能力被拆到 SSO；订单相关逻辑拆到 order；公共 DTO、工具、返回结构等沉淀到 common；服务契约和调用边界由 RPC 层承接。
在这个基础上，系统再按业务需要逐步引入外部能力，比如 Redis 做缓存，Elasticsearch 做商品搜索，支付模块承接交易闭环，短信用于验证码能力，云存储用于资源上传。
所以我会把它描述成一个“按业务能力和用户流程逐步演进的电商平台”，而不是简单的技术大杂烩。

---

## 2. 这个项目怎么划分模块

### 短答版
我会把它分成六类：门户层、后台管理层、身份层、订单层、共享支持层、服务契约层。然后 Redis、搜索、支付、短信、云存储这些能力更像支撑型集成能力，不完全等同于核心业务域。

---

## 3. 为什么要把 portal 和 manager 分开

### 短答版
因为两者面对的用户和流程完全不同。portal 面向终端用户，关注商品展示、下单流程和交互体验；manager 面向运营或后台管理，关注商品管理、内容维护、订单管理等后台流程。虽然都属于 Web 系统，但边界不同，长期混在一起容易让代码职责失控。

---

## 4. 为什么 SSO 要单独做

### 短答版
因为登录、会话、身份校验这类能力通常会被多个业务模块复用，不适合散落在 portal 或其他业务模块里。把它单独做成 SSO，更有利于统一身份处理和后续扩展。

---

## 5. 你怎么讲 Redis / ES / 支付 / 短信这些能力

### 短答版
我不会把它们说成独立主角，而是会说它们是支撑业务流程的能力附件。比如 Redis 解决热点数据访问和缓存问题，ES 解决商品搜索，支付负责交易闭环，短信负责验证链路。重点是它们为什么在项目某个阶段引入，而不是只说“项目用了很多技术”。

---

## 6. 项目连接句

- “这个项目比较像一个按真实购物流程逐步演进的平台，而不是一次性把所有能力都堆上去。”
- “Redis、搜索、支付、短信这些能力我会放在业务流程背景下解释，而不是孤立介绍。”
