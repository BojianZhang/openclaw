# 电商项目追问树

当用户已经能介绍电商项目，但还需要继续接追问时，优先使用这份追问树。

## 根问题：你做过什么电商项目？

### 分支 A：如果是 Ebuy
追问重点：
- 为什么它比基础项目更完整？
- 商品分类、搜索、详情、购物车、订单、后台管理怎么形成完整购物链？
- 为什么购物车和订单值得单独提？
- 为什么它不能被讲成复杂平台？

### 分支 B：如果是 乐SHOP
追问重点：
- 它的模块边界怎么讲？
- portal / manager / sso / order / common / rpc 为什么要拆？
- Redis、ES、支付、短信、云存储这类支撑能力怎么落位？
- 为什么它更适合讲平台能力演进？

### 分支 C：如果是 易购商城
追问重点：
- 为什么它更适合讲传统 Java 分布式商城？
- Dubbo / ZooKeeper / Solr / 购物车 / 订单支付怎么串起来？
- 为什么 RPC 要单独成层？
- 为什么它和 Spring Cloud 路线不同？

### 分支 D：如果是 双11秒杀方案
追问重点：
- 为什么它必须单独讲一条高并发链？
- Redis / MQ / 限流 / 防刷 / 压测分别负责什么？
- 为什么不能拿普通商城样本硬答秒杀？
- 反超卖和削峰怎么解释？

## 二级判断
### 如果面试官继续问普通购物流程
- 优先回到 Ebuy / 乐SHOP / 易购商城

### 如果面试官继续问平台边界和支撑能力
- 优先回到 乐SHOP / 易购商城

### 如果面试官继续问 Dubbo / RPC / 注册中心
- 优先回到 易购商城

### 如果面试官继续问高并发 / 削峰 / 防刷
- 优先回到 双11秒杀方案

### 如果面试官继续问电商项目共性
统一回答：
- Ebuy 偏单体购物链
- 乐SHOP 偏模块化平台能力
- 易购商城 偏传统分布式商城
- 秒杀方案 偏高并发交易链
