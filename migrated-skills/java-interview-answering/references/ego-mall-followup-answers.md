# 易购商城项目追问模板

当用户已经能介绍传统 Java 多模块商城项目，但还需要接更深追问时，优先使用这份模板。

## 1. 为什么用 Dubbo + ZooKeeper

### 回答思路
因为这个项目采用的是传统 Java 多模块商城的服务协作方式，Dubbo 更适合表达服务调用契约和模块间协作边界，ZooKeeper 则承担协调/注册支撑。重点不是“用了 Dubbo 很高级”，而是它帮助把 portal、sso、order 等模块之间的协作边界讲清楚。

---

## 2. 为什么要有 ego-rpc 这一层

### 回答思路
因为它让服务契约和具体实现分开。这样不同模块依赖的是契约，而不是直接依赖彼此的具体实现，长期来看边界更稳，也更适合 Dubbo 这种调用方式。

---

## 3. 为什么 Solr 不是一开始就上

### 回答思路
因为搜索能力属于用户体验优化层，应该在商品和门户基础能力已经比较稳定之后再引入。否则平台基础还没跑稳，就先把搜索体系复杂化，收益并不高。

---

## 4. 为什么购物车不能只当成前端功能

### 回答思路
因为购物车本质上保存的是用户的购买意图状态，它连接商品浏览和订单创建，是一个真实的后端状态能力。如果只当成前端页面功能，很容易忽略数据结构、状态一致性和后续订单链衔接问题。

---

## 5. 为什么订单先于支付

### 回答思路
因为平台要先有内部订单真相，支付结果才能有明确落点。先把订单能力稳定下来，再接支付，会比先接支付 SDK 但内部订单状态还不清晰更合理。

---

## 6. 为什么 RabbitMQ 后置引入更合理

### 回答思路
因为 MQ 的价值在于解耦和异步化，而不是项目一开始就把架构做复杂。像注册通知、邮件发送、消息处理这类流程，在业务路径明确之后再用 MQ 解耦，收益更真实。

---

## 7. 项目答辩连接句
- “我会先讲模块边界，再讲 Dubbo、Redis、Solr、MQ 这些能力为什么在那个阶段引入。”
- “这个项目的重点不在于中间件数量，而在于 portal、manager、sso、order、rpc 这些边界是否清晰。”
- “如果面试官继续追问，我会重点讲为什么购物车、订单、支付要按这种顺序长出来。”
