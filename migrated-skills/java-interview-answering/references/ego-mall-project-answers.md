# 易购商城项目答题模板

当用户要讲一套传统 Java 多模块商城项目，尤其包含 ego-common / ego-manager / ego-portal / ego-rpc / ego-sso / ego-order，以及 Dubbo、ZooKeeper、Solr、Redis、购物车、订单支付、RabbitMQ 这类能力时，优先使用这份模板。

## 1. 你做过什么商城项目

### 短答版
我做过一个传统 Java 多模块商城项目，整体上按能力拆成后台管理、前台门户、公共支持、RPC 契约、单点登录和订单模块。项目最开始先把后台商品管理和前台门户跑通，后面再逐步接入 Dubbo + ZooKeeper 做模块协作、Redis 做缓存和状态加速、Solr 做搜索、SSO 做统一身份、购物车和订单能力、支付能力，以及后面的注册验证和 RabbitMQ 消息处理。这个项目更像一个按业务流程逐步长能力的商城平台，而不是一开始就堆满中间件。

### 展开版
如果让我介绍，我会先从模块边界讲。ego-manager 负责后台商品和运营管理，ego-portal 负责用户侧门户和购物流程，ego-common 放公共 DTO 和工具，ego-rpc 负责服务契约和模块协作边界，ego-sso 负责身份和单点登录，ego-order 负责订单相关能力。
在这个基础上，系统再逐步引入不同的支撑能力：Dubbo + ZooKeeper 用来支撑模块间服务协作，Redis 用于缓存和高频状态访问，Solr 用于搜索体验，购物车承接用户购买意图，订单和支付完成交易闭环，后面再补注册验证、邮件、极验、RabbitMQ 等支撑能力。
所以我会把它描述成一个“按功能和业务流程逐步扩展的商城项目”，而不是单独罗列用了哪些技术栈。

---

## 2. 为什么要单独有 ego-rpc

### 回答思路
因为在这种 Dubbo + ZooKeeper 的商城项目里，RPC 不只是一个依赖，而是服务契约边界。单独有 ego-rpc，能把调用接口和实现边界分开，避免 portal、sso、order 这些模块互相直接耦合具体实现。

---

## 3. 为什么 Solr 是后面才引入

### 回答思路
因为搜索能力通常是在平台和商品基础能力已经具备之后，才开始变成用户体验瓶颈。先把平台、商品、前台页面跑通，再引入 Solr 做搜索，会比一开始就把搜索体系做复杂更合理。

---

## 4. 为什么购物车值得单独讲

### 回答思路
购物车不是一个纯前端功能，它本质上承接的是用户的购买意图状态，介于商品浏览和订单创建之间。所以它既有用户态，又有交易前状态管理的味道，后端也应该把它看成一个明确能力，而不是页面上的一个按钮逻辑。

---

## 5. 为什么订单和支付要分开

### 回答思路
订单是平台内部的交易真相，负责订单创建、状态流转和查询；支付是外部支付能力接入，负责把外部支付结果映射回平台内部订单状态。两者高度相关，但不是同一个边界。如果混着讲，就会把内部业务状态和外部结算过程混在一起。

---

## 6. 为什么 RabbitMQ 是后面才引入

### 回答思路
因为消息队列通常是为了解耦和异步化某些已经存在的业务流程，比如注册通知、邮件发送、消息处理等。它应该在业务流程已经清晰、同步调用开始显得不合适时再引入，而不是项目一开始就为了“显得高级”硬上。

---

## 7. 项目连接句
- “我会把这个项目讲成一个传统 Java 多模块商城按能力逐步演进的过程，而不是技术栈清单。”
- “ego-rpc、ego-sso、ego-order 这些模块是我讲这个项目时最核心的边界。”
- “Redis、Solr、支付、RabbitMQ 这些能力我会放在业务流程和演进时机里讲，而不是孤立介绍。”
- “这个项目最值得讲的是边界和演进顺序，而不是单独用了多少中间件。”
