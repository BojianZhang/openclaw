# Java 项目题 10 题实战演练

当用户要练 Java 后端项目题、项目对比题、样本选择题时，优先使用这份演练集。

## 使用方式
每题都按下面顺序答：
1. 先判断问题域
2. 选最合适的样本
3. 给一句答题总领句
4. 再展开 3-5 个关键点

## 1. 如果面试官问：Spring Cloud 组件那么多，你怎么理解这套体系？
- 最佳样本：Spring Cloud 全家桶
- 总领句：我会把 Spring Cloud 讲成微服务治理组件集合，而不是一串零散名词。
- 关键点：
  - 注册发现、调用、容错、网关、追踪、配置中心分层
  - 组件要放到同一条调用链理解
  - 第一代/第二代演进
  - 组件职责比死记组件名更重要

## 2. 如果面试官问：一个微服务项目怎么逐步长出来？
- 最佳样本：乐Z家
- 总领句：我会按版本演进讲，而不是直接罗列最终模块。
- 关键点：
  - 先平台/共享层，再业务能力
  - 推荐、详情、评论、登录、订单逐步进入
  - Redis/MQ 这类能力后置出现
  - 每次扩展都对应业务压力

## 3. 如果面试官问：秒杀系统怎么设计？
- 最佳样本：双11秒杀方案
- 总领句：我会把秒杀讲成高并发请求处理链，而不是单个扣库存接口。
- 关键点：
  - 入口保护
  - 热点削减
  - Redis 快路径
  - MQ 削峰
  - 最终订单落库与压测验证

## 4. 如果面试官问：为什么电商平台要拆 portal、manager、sso、order？
- 最佳样本：乐SHOP，次选 易购商城
- 总领句：这些边界更多由角色和流程决定，而不是由中间件决定。
- 关键点：
  - portal 面向用户
  - manager 面向运营
  - sso 是身份能力
  - order 是交易真相
  - common/rpc 是支撑边界

## 5. 如果面试官问：为什么要单独做 rpc 层？
- 最佳样本：易购商城
- 总领句：rpc 层的价值在于把契约和实现分开，稳定模块协作边界。
- 关键点：
  - Dubbo + ZooKeeper 协作方式
  - ego-rpc 是协作缝，不是业务实现层
  - portal/sso/order 不直接依赖彼此实现

## 6. 如果面试官问：Solr 和 Elasticsearch 在项目里怎么讲？
- 最佳样本：易购商城 讲 Solr；乐SHOP 讲 ES
- 总领句：两者都应该先讲成搜索体验能力，而不是事务真相存储。
- 关键点：
  - Solr/ES 都服务搜索
  - DB 仍是事务真相
  - 差异是技术路线和时代感

## 7. 如果面试官问：为什么购物车不能只当成前端功能？
- 最佳样本：易购商城
- 总领句：购物车本质上是用户购买意图状态，不只是页面按钮。
- 关键点：
  - 连接商品浏览和订单创建
  - 需要后端状态管理
  - 高并发/高频访问属性明显

## 8. 如果面试官问：为什么订单和支付要分开讲？
- 最佳样本：乐SHOP / 易购商城
- 总领句：订单是内部交易真相，支付是外部结算邻接层。
- 关键点：
  - order truth
  - payment callback/result mapping
  - 两者紧密相关但边界不同

## 9. 如果面试官问：为什么 RabbitMQ 不是一开始就上？
- 最佳样本：双11秒杀方案 / 易购商城
- 总领句：MQ 应该在异步解耦需求已经真实出现时再引入，而不是为了显得高级。
- 关键点：
  - 秒杀里是削峰
  - 易购里是注册/通知/异步工作流
  - 后置引入比先上再找场景更合理

## 10. 如果面试官问：让我讲一个完整电商平台项目，该拿哪套最合适？
- 最佳样本：乐SHOP，次选 易购商城
- 总领句：我会先根据面试官更想听“平台能力演进”还是“传统商城路线”来选样本。
- 关键点：
  - 乐SHOP 更适合模块化平台演进
  - 易购更适合传统 Dubbo 商城
  - 都先讲边界，再讲技术

## 总提醒
- 先匹配问题域，再选样本
- 先讲边界/流程，再讲技术
- 不要拿最复杂的样本去答最窄的问题
