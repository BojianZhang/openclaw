# Java 项目样本真实问题回放

当用户问真实项目题、架构题或面试题时，优先用这份回放集判断“该选哪套样本”以及“最佳回答姿势”。

## 1. Spring Cloud 组件那么多，哪套项目最适合拿来讲？
- 最佳样本：**Spring Cloud 全家桶**
- 原因：它是组件职责、调用链、模板、路线图最集中的样本
- 最佳回答姿势：按组件职责/调用链/代际演进来讲，不要硬拿完整业务项目去解释组件课程题

## 2. 一个微服务项目怎么一版一版长出来？
- 最佳样本：**乐Z家**
- 原因：它最像“按版本逐步长服务”的真实项目样本
- 最佳回答姿势：按阶段讲模块增长和能力后置进入，不要只讲技术点

## 3. 秒杀系统怎么设计？
- 最佳样本：**双11秒杀方案**
- 原因：问题域高度聚焦在高并发交易链
- 最佳回答姿势：按入口保护 -> 热点削减 -> 写路径控制 -> 最终落库来讲

## 4. 一个电商平台为什么要拆 portal / manager / sso / order？
- 最佳样本：**乐SHOP**，次选 **易购商城**
- 原因：两者都有稳定边界，但乐SHOP更适合讲平台能力边界，易购商城更适合讲传统商城结构
- 最佳回答姿势：先讲角色与流程，再讲技术选型

## 5. 为什么要单独有 rpc 层？
- 最佳样本：**易购商城**
- 原因：`ego-rpc` 在传统 Dubbo + ZooKeeper 路线里最有代表性
- 最佳回答姿势：强调契约层和实现层分离，RPC 是协作边界，不是业务实现副本

## 6. Solr 和 Elasticsearch 在项目里怎么讲？
- 最佳样本：**易购商城** 讲 Solr，**乐SHOP** 讲 ES
- 原因：两套项目分别代表传统搜索与较新的搜索能力接入路线
- 最佳回答姿势：都先讲“搜索体验能力”，不要讲成事务真相替代品

## 7. 为什么订单和支付要分开？
- 最佳样本：**乐SHOP** 或 **易购商城**
- 原因：两套平台都能说明 order 是内部交易真相，payment 是外部结算邻接层
- 最佳回答姿势：先讲 order truth，再讲 payment callback / settlement adjacency

## 8. 为什么 RabbitMQ 不是一开始就引入？
- 最佳样本：**双11秒杀方案** 或 **易购商城**
- 原因：秒杀能讲削峰，易购商城能讲注册/通知类异步工作流后置引入
- 最佳回答姿势：强调 MQ 解决已有异步/解耦问题，而不是装饰架构

## 9. 为什么购物车不能只当成前端功能？
- 最佳样本：**易购商城**
- 原因：它明确把购物车作为后端能力阶段单独长出来
- 最佳回答姿势：把购物车讲成用户购买意图状态，而不是页面按钮

## 10. 如果面试官让我讲一个“完整电商平台项目”，该选哪套？
- 最佳样本：**乐SHOP**
- 次选：**易购商城**
- 原因：乐SHOP更适合讲模块化平台能力演进；易购商城更适合讲传统 Java 商城路线
- 最佳回答姿势：根据面试氛围选“更现代的平台演进”还是“更传统的商城实现路线”

## 总原则
- 先匹配问题域，再选样本
- 不要拿最复杂的样本去回答最窄的问题
- 不要把高并发秒杀样本硬套到普通商城平台题上
- 不要把组件课程样本当成完整项目演进样本
