# Java 项目与样本库映射指南

当用户想把“自己的真实项目”映射到现有样本库时，优先使用这份指南。

## 目标
不是强行说自己的项目就是某一套样本，而是回答三个问题：
1. 我的项目更像哪一类样本？
2. 我可以借哪套样本的话术来讲？
3. 哪些样本不应该乱套到我的项目上？

## 第一步：先判断问题域
问自己：
- 我现在更像在讲 Spring Cloud 组件和调用链？
- 还是在讲项目版本演进？
- 还是在讲高并发交易链？
- 还是在讲模块化电商平台边界？
- 还是在讲传统 Dubbo 商城结构？

## 第二步：找最像的主样本

### 如果你的项目更像
- 组件和治理问题 -> 对齐 **Spring Cloud 全家桶**
- 项目分阶段长服务 -> 对齐 **乐Z家**
- 高并发秒杀/抢购 -> 对齐 **双11秒杀方案**
- 模块化电商平台能力演进 -> 对齐 **乐SHOP**
- 传统 Java 商城 + Dubbo/Solr -> 对齐 **易购商城**

## 第三步：允许“主样本 + 辅样本”组合
很多真实项目不会只像一套样本。
一个实用讲法是：
- 主样本决定主叙事
- 辅样本补某个专项点

### 例子
- 你的项目是模块化电商平台，但有秒杀链路 -> 主样本用 **乐SHOP**，高并发部分借 **双11秒杀方案**
- 你的项目是 Spring Cloud 微服务，但也经历了模块渐进增长 -> 主样本用 **Spring Cloud 全家桶**，项目演进部分借 **乐Z家**
- 你的项目是传统 Dubbo 商城，但搜索做得很多 -> 主样本用 **易购商城**，搜索能力可类比 **乐SHOP/ES** 的讲法，但不要混成同一路线

## 第四步：明确不能乱套的部分
- 不要拿秒杀样本替代普通商城平台主叙事
- 不要拿组件课程样本冒充完整业务项目演进
- 不要把传统 Dubbo 商城硬讲成 Spring Cloud 平台
- 不要把乐SHOP 和 易购商城 的技术路线差异抹平

## 第五步：输出时的推荐结构
一个稳的说法是：
1. 先说我的项目更接近哪类样本
2. 再说我借用了哪套样本来组织表达
3. 最后说明有哪些差异，避免让人以为完全照搬

### 参考句式
- “我的项目如果按样本库去对照，更接近模块化电商平台路线，所以我会优先按乐SHOP那条逻辑去讲。”
- “但我项目里有高并发抢购链路，这部分我会借双11秒杀方案的写法去补充。”
- “我不会硬把自己的项目讲成传统 Dubbo 商城，因为技术路线并不一样。”

## 最终原则
- 先求问题域匹配
- 再求表达顺手
- 最后再求样本丰富
- 样本是辅助讲清项目，不是替代你的真实项目
