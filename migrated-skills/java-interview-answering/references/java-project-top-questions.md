# Java 项目样本高频问题集

当用户要准备 Java 后端项目题、架构题、项目对比题时，优先使用这份高频问题集。

## 1. Spring Cloud 组件那么多，拿哪套样本讲最好？
- 最佳样本：Spring Cloud 全家桶
- 回答姿势：按组件职责、调用链、代际演进讲

## 2. 一个微服务项目怎么逐步长出来？
- 最佳样本：乐Z家
- 回答姿势：按阶段讲模块增长和能力后置进入

## 3. 秒杀系统怎么设计？
- 最佳样本：双11秒杀方案
- 回答姿势：入口保护 -> 热点削减 -> 写路径控制 -> 最终落库

## 4. 为什么电商平台要拆 portal / manager / sso / order？
- 最佳样本：乐SHOP，次选 易购商城
- 回答姿势：先讲用户/运营/身份/交易角色，再讲技术

## 5. 为什么要单独有 rpc 层？
- 最佳样本：易购商城
- 回答姿势：讲契约层和实现层分离，强调协作边界

## 6. Solr 和 Elasticsearch 在项目里怎么讲？
- 最佳样本：易购商城 讲 Solr；乐SHOP 讲 ES
- 回答姿势：都先讲“搜索体验能力”，不要讲成事务真相替代品

## 7. 为什么购物车不能只当成前端功能？
- 最佳样本：易购商城
- 回答姿势：把购物车讲成用户购买意图状态和后端状态能力

## 8. 为什么订单和支付要分开？
- 最佳样本：乐SHOP / 易购商城
- 回答姿势：order 是内部交易真相，payment 是外部结算邻接层

## 9. 为什么支付不是一开始就做？
- 最佳样本：乐SHOP / 易购商城
- 回答姿势：先有稳定 order truth，再接外部支付闭环

## 10. 为什么 RabbitMQ 不是一开始就引入？
- 最佳样本：双11秒杀方案 / 易购商城
- 回答姿势：强调解耦和异步化应该在业务路径明确后引入

## 11. Redis 在电商平台里到底承担什么角色？
- 最佳样本：乐SHOP / 易购商城
- 回答姿势：按 portal、sso、order、cart 等不同流讲多角色支撑层

## 12. 为什么 SSO 要单独做？
- 最佳样本：乐SHOP / 易购商城
- 回答姿势：强调身份/session/token 复用和统一治理

## 13. 哪套更适合讲传统 Java 商城？
- 最佳样本：易购商城
- 回答姿势：突出 Dubbo+ZooKeeper、Solr、购物车、订单支付链

## 14. 哪套更适合讲模块化电商平台能力演进？
- 最佳样本：乐SHOP
- 回答姿势：突出 portal/manager/sso/order 边界与 Redis/ES/支付/短信/云存储的阶段引入

## 15. 哪套更适合讲高并发交易链？
- 最佳样本：双11秒杀方案
- 回答姿势：突出 Redis 快路径、MQ 削峰、反超卖、限流、防刷、压测

## 16. 哪套更适合讲传统商城 + RPC 协作？
- 最佳样本：易购商城
- 回答姿势：突出 ego-rpc、Dubbo+ZooKeeper、契约与实现边界

## 17. 哪套更适合讲项目样本库里的“平台边界稳定性”？
- 最佳样本：乐SHOP + 易购商城 对照
- 回答姿势：先讲共同稳定边界，再讲技术路线差异

## 18. 哪套更适合讲“技术不要过早引入”？
- 最佳样本：乐SHOP / 易购商城 / 双11秒杀 组合
- 回答姿势：按“能力压力出现后再引入”的逻辑讲

## 19. 如果面试官让我讲一个完整电商平台项目，该选哪套？
- 最佳样本：乐SHOP，次选 易购商城
- 回答姿势：根据面试氛围选择“更现代的平台演进”或“更传统的商城路线”

## 20. 如果面试官问我为什么用这套样本回答？
- 最佳回答姿势：强调是按问题域选最贴近的样本，而不是按项目大小盲选

## 总原则
- 先判断问题域，再选样本
- 先讲边界/流程，再讲技术
- 不要拿最复杂的样本去答最窄的问题
- 不要把所有项目都讲成一套话术
