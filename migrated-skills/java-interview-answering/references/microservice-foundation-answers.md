# 微服务基础面试答题模板

当题目是“微服务是什么 / 为什么这样设计 / 和其他概念怎么比”时，优先使用这份模板。
如果问题已经进入 Spring Cloud 平台总览，转 `spring-cloud-foundation-answers.md`。
如果问题已经进入具体 Spring Cloud 组件，转 `spring-cloud-components-answers.md`。

## 1. 什么是微服务

### 短答版
微服务是一种架构风格，本质上是把一个大系统按业务能力拆成一组小服务。每个服务都可以独立部署、独立演进，通常通过 HTTP 或消息队列这类轻量通信机制协作。它的优点是边界更清晰、团队自治更强，但代价是分布式系统复杂度会明显上升，比如治理、监控、调用链、事务和容错都会更复杂。

### 展开版
我一般会把微服务定义成“围绕业务能力拆分、可以独立部署、通过轻量通信协作的一组服务”。
它出现的背景是单体系统在规模变大之后，会遇到代码耦合、团队协作、发布风险和扩展效率的问题。
微服务的核心不是把服务拆小，而是把业务边界、数据边界和团队边界尽量对齐。
它带来的收益是独立发布、按需扩缩容、技术栈选择更灵活。
但它也会引入分布式系统的问题，比如网络不可靠、接口版本兼容、链路追踪、重试风暴、分布式事务等。
所以我的理解是：微服务不是“更高级的单体”，而是用更高的系统复杂度，换更好的组织协作和演进能力。

### 常见追问
- 你觉得服务拆多细才算合适？
- 微服务和分布式有什么区别？
- 微服务一定要上 Spring Cloud 吗？

### 易答崩点
- 只讲优点，不讲复杂度
- 把“服务小”当成核心，而忽略业务边界
- 把通信方式当成微服务定义本身

## 2. 单体、分布式、SOA、微服务怎么理解

### 短答版
我会把它看成一个演进过程：单体是所有能力都在一个应用里，分布式是把公共能力拆出去形成跨机器协作，SOA 更强调企业级服务复用和统一治理，微服务则是在 SOA 思想上进一步细化服务边界，强调独立部署、自治和按业务能力拆分。

### 展开版
单体适合早期，开发和部署简单，但规模起来后会变得很重。
分布式的核心是系统跨进程、跨机器协作，不一定已经做到了微服务。
SOA 更偏企业集成和服务复用，常见特点是统一治理、共享规范，甚至会有 ESB 这类中心化中枢。
微服务可以理解成更细粒度、更强调边界自治的一种现代服务化实现方式。
两者都属于面向服务思想，但微服务更强调 bounded context、独立数据、独立部署和团队自治。
所以我不会简单说 SOA 过时、微服务先进，而是会说它们关注点和治理模型不同。

## 3. SOA 和微服务的区别

### 短答版
SOA 和微服务本质上都属于面向服务思想，但 SOA 更强调企业级复用和统一治理，服务粒度通常更粗；微服务强调按业务能力拆分、独立部署、去中心化自治和服务私有数据。可以理解成微服务是更细粒度、更适合云原生和 DevOps 组织方式的一种服务化实践。

### 展开版
我一般从四个维度回答。
第一是粒度，SOA 通常更粗，微服务更细，但细不是目的。
第二是治理方式，SOA 更强调统一标准、统一平台，微服务更强调团队自治和轻治理。
第三是数据边界，SOA 比较容易共享数据库或共享领域模型，微服务更强调每个服务对自己数据负责。
第四是通信方式，SOA 常见 ESB、企业级协议和集中式编排，微服务更倾向 HTTP、RPC、MQ 这种更轻量的方式。
所以我会说：微服务不是否定 SOA，而是把服务化思想往更细粒度、更适合快速迭代的方向推进了一步。

## 4. Dubbo 和 Spring Cloud 的区别

### 短答版
Dubbo 更偏 RPC 调用框架，核心优势是高性能远程调用和接口式服务治理；Spring Cloud 更像一整套微服务生态，除了服务调用，还覆盖注册发现、配置中心、网关、链路追踪、消息总线等能力。简单说，Dubbo 更偏“怎么调远程服务”，Spring Cloud 更偏“怎么搭一个完整的微服务体系”。

### 展开版
如果从定位上看，Dubbo 的核心是 RPC，它强调像调用本地方法一样去调用远程服务，所以在 Java 体系里用起来会比较顺手，性能上通常也会更强一些。
Spring Cloud 的思路更偏平台化，除了调用之外，它还把配置、注册发现、网关、熔断限流、链路追踪这些微服务配套能力整合进来了。
如果从协议风格上看，Dubbo 更偏 RPC，Spring Cloud 常见是 HTTP/REST 风格。
如果从工程治理看，Dubbo 更适合强调高性能 RPC 的 Java 内部服务场景；Spring Cloud 对 Web API、跨团队协作和 Spring 生态整合更友好。
但本质上不管选哪套，分布式里的超时、重试、幂等、版本兼容、治理复杂度都躲不掉。

### 选择建议一句话
如果团队核心诉求是 Java 内部高性能 RPC，可以优先考虑 Dubbo；如果更看重完整微服务生态和 Spring 体系整合，Spring Cloud 更自然。

## 5. 为什么微服务强调无状态

### 短答版
微服务强调无状态，核心是为了让服务实例更容易水平扩展、故障替换和弹性调度。如果状态放在节点内存里，就会带来会话粘连、扩容困难和节点切换问题。所以更常见的做法是把 Session、缓存或上下文状态迁到 Redis 这类外部存储，让业务服务本身尽量只做计算。

### 展开版
这里的无状态不是说系统里没有状态，而是说不要把请求正确性依赖在某一台机器的本地内存上。
如果状态存在服务节点内部，比如本地 Session、本地缓存，那扩容、缩容、重启和故障切换都会变复杂。
所以微服务里常见做法是把状态抽到外部，比如 Redis、数据库或者专门的状态服务里，让业务服务尽量变成无状态计算节点。
这样做的好处是更容易做负载均衡、弹性伸缩和故障恢复。
但代价是外部状态存储会成为新的依赖，所以也要考虑一致性、性能和可用性。

## 6. 为什么微服务常用 RESTful 风格

### 短答版
微服务常用 RESTful 风格，主要是因为 HTTP 生态成熟、跨语言兼容性好、调试成本低，而且 JSON 这类报文人和机器都容易读。它很适合跨团队、跨语言、面向 Web 的服务协作场景。但 REST 不是唯一答案，在高性能内部调用场景下，RPC 或消息队列也很常见。

### 展开版
我会把 RESTful 的优势概括成四点：
第一，HTTP 本身是成熟协议，配套设施像网关、代理、TLS、监控都很完善。
第二，JSON 报文可读性强，联调和排查都比较方便。
第三，跨语言支持很好，不会像某些强绑定 RPC 那样更依赖统一技术栈。
第四，它对前后端分离、多终端访问也比较自然。
但我不会把 REST 说成绝对更优，因为在高吞吐低延迟的内部服务调用里，RPC 也可能更合适；而在解耦异步场景下，消息队列更好。
所以本质不是 REST 更先进，而是它在工程协作和生态成熟度上常常更占优。

## 7. AKF 拆分原则是什么

### 短答版
AKF 立方体是一个系统扩展模型，分成 X、Y、Z 三个方向。X 轴是水平复制，同一个服务多实例加负载均衡；Y 轴是按业务能力拆分，也就是常说的微服务拆分；Z 轴是按用户、地区、租户这类维度做数据分区。它的价值在于提醒我们：微服务只是扩展手段之一，不要一有性能问题就先拆服务。

### 展开版
AKF 我一般会从“扩展系统有三种主要方式”来讲。
X 轴是最直接的方式，就是把同一个应用复制多份，通过负载均衡分摊流量。
Y 轴是按业务能力拆分，比如把订单、支付、库存拆成独立服务。
Z 轴是按数据或用户维度切分，比如按地区、租户、用户分片。
这个模型的价值不是背概念，而是帮助做架构决策：
如果问题只是流量上来了，很多时候先做 X 轴就够；
如果是业务边界和团队协作问题，再考虑 Y 轴；
如果是数据规模或地域性问题，再考虑 Z 轴。
所以它能避免“把服务拆分当成万能药”。

## 8. 前后端分离和微服务是什么关系

### 短答版
前后端分离和微服务不是一回事。前后端分离解决的是界面层和服务层的协作问题，让前端和后端通过 API 解耦；微服务解决的是后端内部按业务能力拆分和独立演进的问题。前后端分离可以和微服务一起出现，但前后端分离本身并不等于微服务。

### 易错提醒
- 不要把“用了 Vue/React + 接口”直接等同于微服务
- 不要把前后端分离说成后端拆服务的依据本身

## 项目化表达补句

当面试官继续追问时，可以补一句：
- “如果结合项目，我会重点讲我们为什么拆、拆完之后怎么治理，以及遇到的超时、重试、配置、链路追踪这些实际问题。”
- “如果项目里还没真正做过微服务，我会明确说明自己掌握的是设计原则、常见治理组件和取舍逻辑，而不是硬编项目经历。”
