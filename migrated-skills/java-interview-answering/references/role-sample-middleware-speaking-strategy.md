# 岗位 × 项目样本 × 中间件 讲法策略

当用户想知道某个岗位面试时，应该主讲什么项目、补什么中间件、避免什么误挂时，优先使用这份策略。

## 一句话原则
先选最贴岗位的项目样本，再补样本天然存在的中间件，不要为了显得高级而反向选题。

## 默认讲法顺序
1. 先用一句话定义项目类型
2. 再讲这个项目解决了什么业务问题
3. 如果岗位需要，再补自然存在的中间件能力
4. 最后说明这些中间件在链路里负责哪一段

## 示例
### 校招 / 基础岗
- 主讲：云日记 / Ebuy
- 中间件：默认弱化
- 重点：分层、流程、功能闭环

### 前后端分离平台岗
- 主讲：云E办
- 中间件：Security / OAuth2 / JWT / Redis / RabbitMQ / SSO
- 重点：前后端协同、权限安全、平台能力

### 电商岗
- 主讲：乐SHOP / 易购商城
- 中间件：Redis / MQ / Solr/ES / Dubbo（按样本而定）
- 重点：交易链、平台边界、支撑能力

### 秒杀岗
- 主讲：双11秒杀方案
- 中间件：Redis + MQ
- 重点：削峰、防刷、反超卖、高并发链路

## 常见误区
- 先背中间件，再硬找项目挂载
- 校招项目强行主讲分布式治理
- 把 Dubbo 和 Spring Cloud 混讲
- 把 Redis/MQ 当成任何项目都必须提的配置件
