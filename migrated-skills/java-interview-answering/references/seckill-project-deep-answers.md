# 秒杀项目深答模板

当用户不只是要一句“Redis + MQ 削峰”，而是要把秒杀项目讲成一条完整高并发处理链时，优先使用这份模板。

## 1. 如果让你完整讲秒杀请求链，你怎么讲

### 短答版
我会把秒杀看成一条高并发请求处理链，而不是单个库存接口。先是用户登录和共享 Session，保证身份识别稳定；然后通过页面缓存、静态资源优化减少热点读压力；接着在真正进入秒杀接口前，用隐藏地址、验证码、接口限流先挡掉一批无效流量；进入业务核心后，用 Redis 做热点预判断，比如库存和重复下单检查；高峰请求再通过 MQ 做异步削峰，最终由后端受控地创建订单和扣减库存，用户再通过结果查询或订单详情确认最终状态。

### 展开版
我一般会按“入口保护 -> 热点削减 -> 写路径控制 -> 最终落库”这四层讲。
第一层是入口保护，包括登录态、共享 Session、隐藏秒杀地址、验证码、接口限流。这一层的目标不是直接完成业务，而是先把脚本流量、无效流量和重复请求拦掉。
第二层是热点削减。商品列表、详情页这种读流量通常远大于真正成功购买的人数，所以会通过页面缓存、静态化、资源缓存等方式减少动态请求量。
第三层是写路径控制。真正进入秒杀动作后，我不会让所有请求都直接同步打数据库，而是会先借助 Redis 做库存或重复下单的快速判断，高峰情况下再通过 MQ 把下单动作异步化，从而削峰填谷。
第四层才是最终业务落地，也就是订单创建、库存确认、结果返回。这样讲能体现我理解的是一整条高并发链路，而不是单独使用了某个中间件。

---

## 2. 如果面试官追问：为什么不用同步直接下单

### 回答思路
因为秒杀场景的核心问题是瞬时洪峰。如果所有请求同步打到订单和库存链路，数据库连接、事务、锁竞争都会在很短时间内放大。同步方案不是绝对不能做，但当流量达到秒杀级别时，通常需要把“请求受理”和“最终下单处理”分开，这就是 Redis 预判断和 MQ 削峰的价值。

---

## 3. 如果面试官追问：怎么防止超卖

### 回答思路
我会把超卖控制拆成几层讲：
第一层是把无效请求尽量挡在前面，比如重复请求、明显无库存、非正常流量；
第二层是在 Redis 这种快路径里做库存预判断和重复下单判断；
第三层是在最终订单和库存落库时做权威确认；
第四层是如果引入 MQ，就要考虑幂等和重复消费，避免同一个请求被重复执行。这样答比单纯说“加锁就行”更完整。

---

## 4. 如果面试官追问：秒杀系统最容易被忽略的点是什么

### 回答思路
我觉得最容易被忽略的是保护层和压测。很多人会只讲 Redis、MQ、扣库存，但如果隐藏接口、验证码、限流、共享 Session、压测验证这些没做，系统可能在真正进入库存逻辑之前就已经被打垮了。

---

## 5. 如果面试官追问：你这个项目更偏高并发保护还是强一致交易

### 回答思路
如果项目本身没有实现特别重的分布式一致性机制，我会诚实说它更偏高并发保护和削峰设计，重点在于承接热点流量、减少无效请求、保护库存和订单链路，而不是去硬吹自己做到了特别复杂的分布式强一致交易。这样会更真实，也更符合项目体量。

---

## 6. 项目连接句
- “我会把秒杀拆成入口保护、热点削减、写路径控制、最终落库四层来讲。”
- “我不会把它说成 Redis 和 MQ 的堆砌，而是讲它们分别承担了快路径判断和异步削峰的职责。”
- “如果项目没有做到特别重的分布式事务，我会如实强调它的重点是高并发保护，而不是硬吹强一致。”
- “压测在这个项目里不是附属环节，而是验证优化有没有真正生效的一部分。”
