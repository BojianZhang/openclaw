# Spring Cloud 调用链面试答题模板

当题目不是在问单个组件定义，而是在问“这些组件怎么协作”“一次请求经过哪些层”时，优先使用这份模板。

## 1. 一次 Spring Cloud 请求经过哪些层

### 短答版
如果按一条典型调用链来讲，我会分成几层：第一层是网关入口，负责统一接入、路由和一些通用过滤；第二层是服务发现和实例选择，比如通过注册中心拿到可用实例，再由 Ribbon 或等价机制做客户端负载均衡；第三层是服务调用本身，比如 Feign/OpenFeign 发起 HTTP 调用；第四层是稳定性保护，比如 Hystrix 或 Sentinel 做超时、熔断、降级、流控；第五层是观测能力，比如 Sleuth 把整条链路串起来。这样答比按组件名堆砌更清楚。

### 展开版
如果让我解释 Spring Cloud 组件怎么协作，我不会一个个背名词，而是按请求路径讲。
先是客户端请求进入网关层，网关负责统一入口、路由、鉴权、过滤等边界能力。
然后具体业务服务需要调用下游服务时，会先通过注册中心获取可用实例列表，再由客户端负载均衡组件挑一个实例。
接着通过 Feign/OpenFeign 这类声明式客户端去发起远程调用。
如果调用过程中出现延迟、异常或流量过大，就会进入稳定性保护层，比如 Hystrix 的隔离熔断降级，或者 Sentinel 的流控和系统保护。
最后，为了知道一次请求到底经过了哪些服务、每一跳耗时多少，还会通过 Sleuth 这类链路追踪组件把 Trace 和 Span 串起来。
所以我理解 Spring Cloud 不是一堆散装组件，而是一条完整的分布式调用治理链。

## 2. 网关、注册中心、Feign、熔断这些组件是什么关系

### 短答版
它们不是平级替代关系，而是分布在同一条调用链的不同位置。网关负责入口，注册中心负责服务地址发现，Feign 负责发起调用，Ribbon 这类组件负责挑选实例，Hystrix/Sentinel 负责稳定性保护，Sleuth 负责观测整条链路。

### 易错提醒
- 不要把注册中心说成“负责转发请求”
- 不要把 Feign 说成“自动解决容错问题”
- 不要把网关和 Ribbon 说成同一种负载均衡

## 3. 如果一次调用失败，应该先从哪一层解释

### 短答版
我一般会按路径排：先看有没有走到网关，再看路由是否正确；然后看注册中心拿到的实例是否健康；再看 Feign 或底层 HTTP 调用是否超时；再看是不是被 Hystrix/Sentinel 提前熔断、降级或限流；最后结合 Sleuth 或日志找出第一跳出问题的位置。

### 展开版
面试里如果被问到调用失败怎么分析，我会先强调不要只盯一个组件。
更合理的方式是沿着调用链排查：
第一看入口层，也就是网关有没有路由错、鉴权拦截或者超时。
第二看服务发现层，实例列表是不是正确，注册中心里是否存在已经不健康但还没摘掉的节点。
第三看调用层，Feign/OpenFeign 或底层 HTTP 客户端的连接、读取超时怎么配的。
第四看保护层，失败到底是真正的网络问题，还是 Hystrix/Sentinel 触发了熔断、降级、限流。
第五看观测层，用 Trace / Span 把第一跳慢在哪里找出来。
这样回答能体现你理解的是“系统路径”，不是只会背组件定义。

---

## 4. 为什么说 Spring Cloud 组件要放在调用链里理解

### 短答版
因为很多组件的职责本来就不是孤立的。单独背组件定义容易碎片化，真正做架构设计或排障时，必须知道它在调用链里的位置、前后依赖和边界。

### 一句补充
面试里如果能把组件从“名词题”回答成“路径题”，通常会更像真正做过系统的人。
