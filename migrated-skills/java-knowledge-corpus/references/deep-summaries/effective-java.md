# Effective Java - 深摘要

## 这本书解决什么问题
它解决的不是“Java 语法怎么写”，而是：**怎样写出长期可维护、边界清晰、少踩坑的 Java 代码**。

## 核心主线
1. 正确创建和销毁对象
2. 正确设计类和接口
3. 正确使用泛型、枚举、注解
4. 正确处理异常
5. 在并发和序列化场景下避免隐蔽问题

## 应掌握的关键思想
### 1. API 设计优先于实现细节
- 接口应该清晰表达约束
- 方法签名要稳定、可读、少歧义
- 不要把调用者绑死在实现细节上

### 2. 可变性是复杂度来源
- 能不可变就不可变
- 可变对象会放大并发问题、缓存问题、拷贝问题
- DTO、值对象、配置对象尤其值得优先考虑不可变设计

### 3. 组合优于继承
- 继承会把父类实现细节泄露给子类
- 很多“想复用代码”的场景，本质更适合委托/装饰器

### 4. 尽量让错误尽早暴露
- 参数校验前置
- 非法状态要快速失败
- 不要把错误拖到系统深处才炸

## 工程落地价值
### 适合回答的问题
- Java 类怎么设计更合理？
- equals/hashCode 为什么总出错？
- 什么时候用接口，什么时候用抽象类？
- Builder 模式什么时候值得上？
- 为什么要少暴露 setter？

### 对项目最有用的章节能力
- 静态工厂方法 vs 构造器
- Builder 适用场景
- 覆盖 equals/hashCode/toString 的规范
- 防御性拷贝
- 异常边界设计
- 泛型 API 设计

## 常见误区
- 误以为“写得短就是写得好”
- 误以为“继承就是复用”
- 误以为“getter/setter 全开放最灵活”
- 误以为“异常越吞越稳”

## 学完后应具备的能力
- 能设计更稳定的 Java API
- 能判断一个类是否职责过重
- 能识别可变对象带来的隐患
- 能写出更适合团队维护的 Java 代码

## 与其他书的连接
- 和《代码整洁之道》一起用：前者重设计约束，后者重可读性
- 和《重构》一起用：一个讲目标，一个讲改法
- 和《Java 并发编程实战》一起用：不可变性在并发里价值更大
