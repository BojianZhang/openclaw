# 专题卡：基础项目 vs 复杂项目 选择原则

## 适用问题
- 面试时什么时候该讲基础项目，什么时候该讲复杂项目？
- 为什么有些问题反而更适合用简单样本回答？
- 为什么项目越复杂不一定越适合回答问题？

## 核心原则
**问题域匹配** 和 **表达真实度**，通常比项目复杂度更重要。

## 什么时候优先选基础项目样本

优先选像“云日记”这样的基础项目，当问题更关注：
- 传统 Java Web 开发流程
- Servlet/JSP/DAO 基础分层
- 登录态处理、Session/Cookie
- BaseDao 封装与基础数据库访问
- 模块拆分的基本思路
- 报表/统计、前后端交互、功能闭环

### 典型场景
- “你有没有做过传统 Java Web 项目？”
- “DAO/BaseDao 为什么值得讲？”
- “项目简单时怎么体现基础能力？”
- “Servlet/JSP 项目有什么价值？”

## 什么时候优先选复杂项目样本

优先选复杂样本，当问题更关注：
- 微服务组件治理
- RPC 协作
- 电商平台边界
- 高并发交易链
- Redis/MQ/搜索/支付等支撑能力的引入顺序

### 典型样本
- Spring Cloud 全家桶
- 乐Z家
- 双11秒杀方案
- 乐SHOP
- 易购商城

## 常见误区

### 误区 1：项目越复杂越显得厉害
问题：
- 复杂项目如果问题域不匹配，反而更容易答偏
- 还容易显得像在背技术栈而不是回答问题

### 误区 2：基础项目不值得讲
问题：
- 基础项目最能体现底层功底和开发闭环
- 如果讲得真实，反而更容易建立可信度

### 误区 3：拿复杂项目硬包装基础问题
问题：
- 比如问 BaseDao、Session/Cookie、Servlet 分层，却非要拿电商平台或微服务项目来讲
- 会让回答显得飘、不接地气

## 选择判断公式

可以用一个很简单的判断公式：
- 如果问题在问 **基础开发能力/传统 Web 基础** -> 先选基础项目
- 如果问题在问 **平台边界/组件协作/高并发链路** -> 再选复杂项目

## 推荐表达
- “这类问题我更愿意用基础项目来讲，因为它更能体现底层开发能力。”
- “这类问题如果硬拿复杂项目来答，反而会把问题域带偏。”
- “复杂样本适合讲边界和能力演进，基础样本适合讲开发闭环和基本功。”
