# 专题卡：Java 项目样本误用警告

## 适用问题
- 这些项目样本最容易被怎么讲歪？
- 哪些样本不适合回答哪些问题？
- 为什么会出现“样本看起来很大，但答题反而答偏了”？

## 核心原则
选样本时，**问题域匹配** 比 **样本体量大** 更重要。

## 常见误用

### 1. 用 Spring Cloud 全家桶硬答完整项目演进
问题：
- 这套资料更偏组件职责、模板、路线图
- 它不天然等于完整业务平台演进样本

正确用法：
- 讲组件、调用链、网关、容错、配置中心，用它
- 讲完整项目如何一版一版长出来，优先换成 乐Z家 / 乐SHOP

### 2. 用秒杀样本硬答普通商城平台题
问题：
- 秒杀样本的问题域高度集中在高并发交易链
- 它不适合拿来讲普通商城平台的 portal/manager/sso/order 边界

正确用法：
- 讲 Redis 快路径、MQ 削峰、反超卖、限流、防刷，用秒杀样本
- 讲普通电商平台模块边界，用 乐SHOP / 易购商城

### 3. 用传统 Dubbo 商城硬答 Spring Cloud 组件题
问题：
- 易购商城 很适合讲传统 Java 商城、Dubbo、Solr、购物车、订单支付
- 但它不是解释 Gateway/Feign/Hystrix/Sentinel/Config 的最佳样本

正确用法：
- 讲传统商城架构，用 易购商城
- 讲 Spring Cloud 组件职责，用 Spring Cloud 全家桶

### 4. 用乐SHOP硬答高并发秒杀专项
问题：
- 乐SHOP 更适合讲模块化平台边界和能力演进
- 不适合替代秒杀专项中的热点读写、削峰、反超卖问题域

正确用法：
- 讲平台能力渐进引入，用 乐SHOP
- 讲高并发交易链，用 双11秒杀方案

### 5. 用乐Z家硬答纯组件框架题
问题：
- 乐Z家 更适合讲完整微服务项目怎么长服务
- 不适合作为 Spring Cloud 组件定义题的首选样本

正确用法：
- 讲项目版本演进，用 乐Z家
- 讲组件职责和调用链，用 Spring Cloud 全家桶

### 6. 把所有电商项目讲成一套话术
问题：
- 乐SHOP 和 易购商城 都是电商平台，但技术路线和价值点不同
- 一个偏模块化平台能力演进，一个偏传统 Dubbo 商城链路

正确用法：
- 先讲共享边界，再讲路线差异
- 不要把 Solr/ES、Dubbo/Spring Boot、购物车/支付/MQ 的差异抹平

## 快速自检问题
在选样本前先问自己：
1. 这题是在问组件、项目演进、平台边界，还是高并发交易链？
2. 这套样本最擅长回答这个问题，还是只是“资料很多”？
3. 有没有更贴近问题域的样本？
